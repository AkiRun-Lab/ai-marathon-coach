# マラソントレーニング・プランナー 更新履歴

## v1.12.0 (2026-07-12)

### ✨ 新機能（計画の分析表示と収益導線の強化）

- **週間負荷グラフ**: 計画本文の直後に「📊 週間負荷の推移」を追加。週別走行距離を強度種別（E/M/T/I/R/その他）の積み上げ棒グラフで表示する（Streamlit同梱のAltairを使用・新規依存なし）。強度は青の単一色相・薄→濃で符号化し、ダーク背景に対して色覚多様性・コントラストを検証済み。ホバーで週・種別・距離を表示。キャプションに動的サマリ（合計・週平均・ピーク週）と「前週比10%以内が目安」の注記を併記
- **計画連動シューズ提案CTA**: 計画の練習構成（ポイント練習の週平均回数・週間距離）から point_heavy / balanced / easy_focus を判定し、計画データを引用したパーソナライズ文言のシューズ提案カードを表示。集計が取れない場合は従来の汎用ギアCTAに自動フォールバック（CTAは常に1枚のみ）。送客先はカテゴリ別のAmazonアイデアリスト（point_heavy→レースシューズ、balanced/easy_focus→デイリートレーナー、generalのみストアトップ）

### 🔧 改善

- **計画JSONの集計基盤を新設**（`src/plan_stats.py`）: Geminiが返す計画JSONから週別距離・強度内訳を機械集計する。Gemini API呼び出しは一切増やさない。距離の範囲表記・キー欠落・非数値に防御的で、集計失敗時も計画表示本体には影響しない（グラフ・CTA非表示になるのみ）。強度分類はpace内の強度記号（E/M/T/I/R）を最優先し、menuの記号・キーワード（ジョグ・閾値・クルーズ・Eペース等）でフォールバック。複合練習（例: pace「E 5:33/km, M 4:30/km」）は最も強度の高い要素で分類し、ポイント練習を含む日を取りこぼさない。練習内の「レスト◯分」を休養日と誤判定しない
- `convert_json_to_markdown` の戻り値を `(Markdown, 週数, plan dict)` の3値に拡張（パース済みJSONの再利用。失敗時は `(None, 0, None)`）
- 新規チャートの幅指定に `width="stretch"` を使用（`use_container_width` の非推奨対応）

### 🧪 テスト

- `tests/test_plan_stats.py`・`tests/test_shoe_cta.py`・`tests/test_weekly_load_chart.py` を新規追加（計35件）。実際のGemini出力（12週・85日分）での分類精度も検証済み（分類不能はレース当日のみ）

## v1.11.0 (2026-07-11)

### 🛡️ 堅牢性強化（503多発・APIハング対策）

背景：計画生成に使う gemini-3.5-flash で503（モデル高負荷）が多発。さらにSDKのHTTPタイムアウトはデフォルト無期限のため、APIがハングすると進捗ループが永久に回り続ける問題（watchdog欠落）があった。

- **APIタイムアウトの明示**: `GenerateContentConfig` に `http_options`（`PLAN_TIMEOUT_SEC`=600秒）を設定。ハング時は10分で断念し、`TIMEOUT_EXCEEDED` として分類・「計画生成が10分を超えたため中断しました」と表示する（リトライ・フォールバックはしない）。
- **503時の2段フォールバック**: 選択モデルでのリトライ（計3回・指数バックオフ）が503で尽きた場合のみ、`GEMINI_FALLBACK_MODEL`（gemini-3-flash-preview）へ自動で切り替えて最大2回試行する。フォールバック先はUIのモデル選択とは独立に常にこのモデル（JSONモード＋thinking_level互換はSDT v1.13.0の実プローブ・実運用で確認済み）。429・タイムアウト・その他のエラーではフォールバックしない。代替モデルで生成した計画には画面上とダウンロードMDの先頭に注記を表示する。
- **進捗ループのwatchdog**: 生成が700秒を超えたら進捗ループを打ち切り、エラー状態へ遷移して再生成ボタンを提示（単発ハングは600秒で断念するため、余裕をみて700秒）。
- **進捗表示の改善**: 段階的文言に「n分nn秒経過」を併記。リトライ中は「自動再試行n回目/最大3回」、フォールバック中は「混雑のため代替モデルで生成中...」を表示。失敗時はプログレスバーを消してからエラー表示する（「生成完了！」や「生成中」表示を残さない）。

## v1.10.2 (2026-07-03)

### 🐛 バグ修正（コードレビュー フェーズ1: 生成フローの再設計）
- **エラー後の自動再実行を廃止**: 生成失敗後に画面操作（rerun）のたびAPIが自動で再コールされていた問題を修正。エラー時は状態として保存し、「🔄 再生成する」ボタンからのみ再実行する方式に変更。
- **生成中のAPI二重コールを防止**: 生成中（約90秒）にユーザーが画面を操作すると、実行中のAPIスレッドが放置されたまま新しいAPIコールが開始されていた問題を修正。スレッドと結果をsession_stateに保持し、rerun後は同じスレッドに再接続して進捗表示を継続する。
- **変換エラーが「計画」として保存される問題を修正**: JSON変換失敗時のエラーメッセージ文字列が生成結果としてsession_stateに保存され、再試行手段がないままダウンロードボタンまで表示されていた問題を修正。`convert_json_to_markdown` は失敗時に `None` を返し、呼び出し側でエラー表示＋再生成ボタンを出す。

### 🔧 改善（コードレビュー フェーズ2: API設定・潜在バグ）
- **`include_thoughts=True` を削除**: 思考サマリーは未使用のため受け取らない。JSON応答への思考テキスト混入リスクとレスポンスサイズ増を排除（思考自体は `thinking_level="high"` で従来どおり有効）。
- **最大出力トークンの床値を16384→24576に引き上げ**: thinkingトークンが `max_output_tokens` を消費するため、12週計画（本文約16,400トークン）で途中切断が起きうる問題への対策。
- **VDOT補間の潜在NameErrorを修正**: 参照タイムが同値のとき `ratio` 未定義のまま計算ログで参照してクラッシュする地雷を解消（`src/vdot/calculator.py`）。

### 🔧 改善（コードレビュー フェーズ3: UX・整合性）
- **ニックネームのHTMLエスケープ**: VDOT結果表示（`unsafe_allow_html`）に埋め込むユーザー入力を `html.escape` で無害化。ダウンロードファイル名もファイル名に使えない文字をアンダースコアへ置換。
- **中間目標の説明文をUIと整合**: プロンプト内にハードコードされていた「VDOT差 3.0」を、実際の許容差（走力・期間に応じた動的値）に置換。生成計画の説明とUI警告の食い違いを解消。
- **日付計算を日本時間（JST）基準に統一**: 本番サーバー（UTC）で最大9時間ズレていた「今日」の判定を `jst_now()`（Asia/Tokyo）に統一。開始日の月曜計算・レースまでの週数のズレを解消。あわせて未使用の `is_past_start` を削除。
- **レース日に過去日付を選べないように**: `st.date_input` に `min_value=今日` を設定（週数が負になる異常表示を防止）。
- **リトライを503/429のみに限定**: APIキー不備等の確定失敗まで2回リトライして計3回コール＋約6秒待たせていたのを即時エラー表示に変更。

### 🧹 クリーンアップ（コードレビュー フェーズ4）
- **未使用コードを削除**: `sanitize_gemini_output`（どこからも呼ばれていない）、`src/vdot/training.py`（`get_training_start_date` はapp.py内の実装と重複し未使用）、リポジトリ直下の手動確認スクリプト `test_format.py`。
- **週数チェックのJSON二重パースを解消**: `convert_json_to_markdown` が `(Markdown, 週数)` を返すよう変更し、app.py側の再パース（`lstrip('```json')` の文字集合誤用を含む）を削除。
- **JSON修復の挿入キーを中立化**: `_repair_json` の汎用パターンが欠落キーを `"menu"` と決め打ちし、既存の正しい `menu` を後勝ちで上書きしうる問題を `"_repaired"` キーに変更して解消。
- **requirements.txt から pytest を分離**: 本番（Streamlit Cloud）に不要なテスト依存を `requirements-dev.txt` へ移動。
- **表記の陳腐化を解消**: app.py docstringの固定バージョン表記を削除（`APP_VERSION` を正とする）、フッターの著作権年を動的化（© 2025–現在年）。
